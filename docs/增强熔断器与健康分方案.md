# 增强熔断器与健康分方案

> 目标：在现有负载均衡与熔断机制之上，增强熔断器的长期质量治理能力，降低交互式客户端在多通道聚合场景下的首包延迟、失败重试成本与坏通道污染。
> 更新日期：2026-03-28

---

## 一、背景问题

当前项目已经具备以下能力：

- 多通道聚合
- 负载均衡
- 会话保持
- 熔断器（连续失败后临时跳过）

但现有机制主要解决的是“短时故障隔离”，对“长期质量差但偶尔可用”的通道处理不够。

在交互式客户端场景下，这会导致几个典型问题：

1. 某些通道虽然没有彻底宕机，但成功率低、首包慢、超时多，仍会频繁参与排序。
2. 同一个坏通道如果挂载了多个模型别名，会反复污染候选列表。
3. 熔断器会自动半开探活，导致质量差通道不断重新进入请求路径。
4. 对 `codex` 这类高频、强依赖首包速度的客户端，体感会明显恶化。

本方案的目标不是替代熔断器，而是将其升级为“带健康分与动态冷却的增强型熔断器”，使排序结果能够自动偏向最近真实表现更好的通道。

---

## 二、设计目标

### 2.1 目标

新增一套运行时健康治理机制，满足以下要求：

1. 坏通道能被快速降权，而不是仅在失败阈值达到后才短暂跳过。
2. 排序不仅依赖静态 `priority/weight`，还要参考近期真实表现。
3. 对确定性错误（如 TLS、401、403、模型不存在）应给予更强惩罚。
4. 对短时抖动通道保留恢复机会，避免永久惩罚。
5. 对交互式请求优先优化首包速度与稳定性，而不仅是最终成功率。

### 2.2 非目标

本阶段不追求：

1. 完整的机器学习路由。
2. 跨机器分布式共享健康状态。
3. 复杂的 UI 可视化控制台。

首版优先做“本机单实例、可解释、低侵入”的机制。

---

## 三、现有机制与缺口

### 3.1 现有机制

当前已有以下与质量控制相关的机制：

1. `priority/weight`
   - 静态配置
   - 适合表达人工意图，不反映运行时健康状况

2. `session keep`
   - 成功后会话粘性保持
   - 有助于复用好通道，但前提是先命中过一次好通道

3. `circuit breaker`
   - 连续失败后进入 `Open`
   - 冷却后进入 `HalfOpen`
   - 成功恢复，失败继续打开

### 3.2 缺口

当前缺少：

1. 对“慢但没死”的通道进行降权。
2. 对错误类型进行差异化处理。
3. 将运行时健康状态接入排序。
4. 让熔断器具备更长周期的重惩罚与隔离能力。

---

## 四、总体方案

方案分为三层能力：

1. 健康分（Health Score）
2. 动态冷却（Dynamic Cooldown）
3. 动态排序（Health-aware Ordering）

三者关系如下：

- 健康分负责描述“最近表现如何”
- 动态冷却负责处理“已知质量极差或确定性错误”
- 动态排序负责把健康状态真正转化为请求路径收益

---

## 五、健康分设计

### 5.1 评分对象

建议以 `(channel_id, channel_key_id, model_name)` 为最小粒度维护健康状态。

原因：

1. 不同模型在同一通道上的可用性可能不同。
2. 同一通道不同 key 可能有配额、权限、风控差异。
3. 粒度过粗会误伤同通道下的其它可用模型。

可选的扩展聚合维度：

- `channel_id` 维度的辅助分数
- `channel_id + model_name` 维度的辅助分数

首版不一定要全部实现，优先实现请求路径直接使用的主键粒度。

### 5.2 初始分数

建议每个评分对象初始化为 `0`，表示中性。

分数越高，表示近期更健康；分数越低，表示近期更差。

### 5.3 基础事件加减分建议

建议首版采用固定规则，便于解释与调参：

| 事件 | 建议分值 | 说明 |
|------|---------|------|
| 成功完成请求 | `+3` | 正常恢复通道信任 |
| 成功且首包很快 | `+5` | 可选，首包表现好时加分更多 |
| 普通失败 | `-10` | 5xx、未知转发失败 |
| 首包超时 | `-20` | 交互式场景中影响极大 |
| 连接超时/网络超时 | `-15` | 明显影响体验 |
| TLS 证书错误 | `-50` | 确定性错误，应重罚 |
| 401/403 鉴权失败 | `-50` | 通常短期不可恢复 |
| 404/模型不存在 | `-30` | 对该模型映射几乎无效 |
| 参数不兼容/协议不支持 | `-20` | 质量差但未必是整个通道全坏 |

### 5.4 时间衰减

为了避免通道被永久惩罚，建议引入时间衰减：

1. 分数逐步向 `0` 回归。
2. 长时间无请求的通道恢复到中性，而不是一直保持低分。

建议实现方式：

- 每次读写健康分时，根据 `last_updated_at` 计算衰减
- 例如每 10 分钟向 0 回收 10%

首版也可以先不做连续衰减，而是在后台定时任务中批量衰减。

### 5.5 上下限控制

建议设置分数边界，避免数值无限增长：

- 最低分：`-100`
- 最高分：`100`

## 六、动态冷却设计

### 6.1 定义

动态冷却不是独立于熔断器的新机制，而是增强熔断器的一部分。

当候选出现严重失败或长期低健康分时，熔断器的 `Open` 时长不再仅依赖固定指数退避，而是根据错误类型与健康分动态延长。

### 6.2 触发条件

建议两类触发：

1. 分数触发
   - 当健康分低于阈值，如 `<= -50`

2. 事件触发
   - TLS 证书错误
   - 401/403
   - 明确的模型不存在

### 6.3 时长建议

建议按严重程度分层：

| 场景 | 时长建议 |
|------|---------|
| 普通严重失败累计过多 | 10 分钟 |
| 首包超时反复发生 | 15 分钟 |
| TLS/401/403 | 30 分钟到 1 小时 |
| 模型不存在 | 1 小时或直到配置变更 |

### 6.4 与现有熔断器的关系

建议不要引入独立“小黑屋”状态，而是在现有 `Closed/Open/HalfOpen` 体系中扩展能力：

1. `Closed`
   - 正常参与排序与请求

2. `Open`
   - 暂时不参与请求
   - `Open` 时长根据错误类型和健康分动态调整

3. `HalfOpen`
   - 少量探活请求

4. `HealthScore`
   - 作为熔断器内部属性
   - 用于决定冷却时间和排序惩罚

---

## 七、动态排序设计

### 7.1 目标

将当前纯静态排序升级为“静态配置 + 运行时健康”的混合排序。

### 7.2 排序原则

建议排序分两阶段：

1. 静态阶段
   - 先按已有 `priority/weight/mode` 生成基础候选顺序

2. 动态阶段
   - 在不破坏总体策略语义的前提下，按健康分调整局部顺序

### 7.3 推荐策略

不同模式下建议不同处理：

1. `Failover`
   - 原本按优先级排序
   - 低健康分候选通过惩罚型排序后移
   - 同优先级内允许对长期未采样的健康候选做低频探索，但不跨优先级抢占

2. `RoundRobin`
   - 保留轮询主语义
   - 轮询计数需按分组隔离，避免不同分组互相影响
   - 健康分仅用于降级异常候选，不再让连续成功的候选长期霸榜

3. `Random/Weighted`
   - 保留随机/加权主语义
   - 健康分主要用于后移异常候选，而不是持续抬升已成功候选
   - 对长期未采样且健康的渠道加入低频探索，避免样本长期陈旧
   - 同渠道多 key 的补采样建议保留独立开关，便于对高成本或配额敏感 key 单独关闭
   - 探索频率应可配置；对 10-30 条目的常用分组，默认每 6 次请求探索一次通常更均衡

### 7.4 健康分影响方式

建议不要直接完全覆盖人工优先级，而是采用“惩罚型排序”：

- 健康分高：默认保持基础策略顺序，不再持续前移
- 健康分低：明显后移
- 长冷却中的候选：直接跳过
- 长期未采样但健康的候选：仅在低频探索窗口内获得前置机会

这是因为用户配置的 `priority` 本身通常包含业务意图，不应被完全抹掉。

## 八、建议的数据结构

### 8.1 运行时结构

建议新增类似如下的运行时结构：

```go
type HealthEntry struct {
	ChannelID      int
	ChannelKeyID   int
	ModelName      string
	Score          int
	SuccessCount   int64
	FailureCount   int64
	TimeoutCount   int64
	LastErrorType  string
	LastSuccessAt  time.Time
	LastFailureAt  time.Time
	UpdatedAt      time.Time
	OpenUntil       time.Time
}
```

首版可先放在内存中，与当前熔断器类似。

### 8.2 是否持久化

建议分阶段：

1. 第一阶段
   - 仅内存
   - 快速上线，验证收益

2. 第二阶段
   - 持久化到数据库或缓存
   - 重启后保留一部分近期状态

若做持久化，可考虑新增一张表，例如：

- `channel_health`

字段包含：

- `channel_id`
- `channel_key_id`
- `model_name`
- `score`
- `quarantine_until`
- `last_error_type`
- `updated_at`

## 九、错误分类建议

这是本方案能否有效的关键。

建议至少区分以下错误类型：

1. `tls_error`
   - 例如证书不受信

2. `auth_error`
   - 401、403

3. `model_not_found`
   - 上游模型不存在

4. `first_token_timeout`
   - 首包超时

5. `network_timeout`
   - 连接或读取超时

6. `upstream_5xx`
   - 上游服务异常

7. `protocol_error`
   - 响应格式不兼容、转换失败

8. `success`

错误分类应尽量在一次请求结束时统一归口，这样健康打点逻辑更集中。

## 十、与现有代码的接入点

### 10.1 排序入口

可在以下位置接入健康排序：

- `internal/relay/balancer/iterator.go`
- `internal/relay/balancer/balancer.go`

建议流程：

1. 现有 `Balancer.Candidates()` 先产出基础顺序
2. 在 `NewIterator()` 中应用健康状态重排与长冷却过滤

### 10.2 打点入口

可在以下位置记录成功或失败：

- 请求成功：`relayAttempt.attempt()` 成功分支
- 请求失败：`relayAttempt.attempt()` 失败分支
- 首包超时：`handleStreamResponse()` 中 first token timeout 分支

### 10.3 配置入口

建议新增可配置项：

- 健康分是否启用
- 动态冷却是否启用
- 分数阈值
- 各类错误惩罚值
- 各类错误对应的冷却时长
- 分数衰减周期

这些配置可先进入 `settings` 表，以便后续在管理界面可调。

## 十一、推荐实施阶段

### 阶段一：最小可用版本

目标：快速止损。

实现内容：

1. 增强现有熔断器条目，新增健康分相关字段
2. 对成功、普通失败、首包超时、TLS/401/403 进行打点
3. 新增动态冷却机制
4. 在 `NewIterator()` 中跳过长冷却候选
5. 在同优先级或近邻候选中按健康分排序

收益：

- 坏通道快速后移
- 明确错误快速进入更长冷却
- 风险较低，改动面较小

### 阶段二：增强版

实现内容：

1. 健康分持久化
2. 错误类型更细分
3. 健康分对 `weighted/random` 模式进行权重修正
4. 增加衰减机制

### 阶段三：运维可视化

实现内容：

1. 后台查看每个通道模型的健康分
2. 查看熔断器长冷却状态和剩余时间
3. 手动释放长冷却
4. 手动重置健康分

## 十二、风险与注意事项

### 12.1 风险

1. 打分过重
   - 可能导致偶发抖动通道被过度惩罚

2. 打分过轻
   - 无法有效改善体验

3. 误伤粒度
   - 如果只按 `channel_id` 评分，可能把其它正常模型一起拖下去

4. 与人工优先级冲突
   - 若健康分权重过大，会破坏用户手工配置意图

### 12.2 建议

1. 首版先做“保守降权”，不要完全替代 `priority`
2. 对确定性错误使用强惩罚和更长冷却，对瞬时错误使用中等惩罚
3. 日志中增加健康分变更记录，便于调试
4. 先在 `codex` 高频模型组上验证收益，再推广到所有组

---

## 十三、预期收益

如果该方案实现合理，预期可改善以下问题：

1. 坏通道不会长期停留在候选前列。
2. 同一坏通道挂载多个模型别名时，能更快整体后移。
3. 高交互客户端的首包时间和失败重试次数下降。
4. 对人工配置不完善的模型组有更强的自愈能力。

特别是在 `codex` 这类对首包延迟敏感的使用场景下，增强熔断器与健康分机制相比当前单纯熔断，理论上会更稳定地改善体感。

---

## 十四、结论

建议在当前项目中引入“健康分 + 动态冷却 + 动态排序”的增强型熔断器，而不是仅继续依赖静态优先级与当前短期熔断。

推荐实施顺序：

1. 先增强运行时熔断器并加入健康分
2. 再将健康状态接入排序
3. 最后补持久化与可视化

这是一个投入相对可控、收益较高、且与现有架构兼容度较好的演进方向。
